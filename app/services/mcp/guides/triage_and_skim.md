# Triage and Skim

Campbooks classifies every incoming email into a category. The Skim deck groups
those categories into rings you can work through in one pass.

## Categories

| Category | What ends up here |
|---|---|
| personal | Individual senders the workspace recognises as people |
| important | Emails the AI scores high-importance (urgency, action required) |
| notifications | Automated app notifications (GitHub, Jira, Slack digests, etc.) |
| promotions | Marketing and promotional mail |
| social | Social network updates, invites |
| updates | Newsletters, digests, subscription updates |
| unknown | Unclassified (classifier returned no confident label) |

## Smart groups

Inside the inbox, emails of the same category from the same sender cluster
together into a group. This is visible to agents through the Skim deck —
each cluster card carries `category`, `count`, and a `summary` generated by
the AI so you do not need to fetch every email to form a view.

## Awaiting-reply

`get_overview` returns `awaiting_reply_count` — the number of email threads
where the current user sent the last message and has not heard back. These are
detected without AI (header heuristics only). Useful as a gentle reminder to
follow up.

## The Skim loop

1. **get_skim_deck** — returns rings. Each ring has a `theme` (personal /
   notifications / promotions / etc.) and a list of cluster cards.

2. **Read the clusters** — each cluster gives you:
   - `title` and `summary` (AI-generated, one sentence)
   - `importance` (1–5)
   - `priority_suggested` — true when the AI thinks this cluster should be
     read before others
   - `scout_suggestion` — a one-line action suggestion when Scout has one
   - `follow_up` / `follow_up_reason` — signals a reply may be overdue
   - `email_ids` — the ids of emails in this cluster; first 3 also appear as
     `samples` with from/subject for quick scanning

3. **Propose decisions per cluster** — the valid actions are:
   - `keep` — dismiss from the Skim tray without archiving (you've seen it,
     no action needed)
   - `archive` — move the thread out of the inbox to the archive
   - `promote` — pin the thread to keep it at the top of the inbox
   - `restore` — undo an archive
   - `unpromote` — remove a pin

4. **skim_decide(action, email_ids)** — applies the decision and feeds the
   learning loop. keep/archive/promote also call the decision recorder, which
   trains future triage suggestions. Always confirm with the user before
   bulk-archiving a cluster they have not reviewed.

## Pinned semantics

Promoted emails appear at the top of the inbox with a pin indicator. They
survive inbox cleaning and are not affected by archive sweeps. Use promote for
threads the user explicitly wants to track.

## Working rhythm

A healthy triage session: get_overview (headline numbers) → get_skim_deck
(inspect clusters) → propose one ring at a time → skim_decide for confirmed
batches → check awaiting_reply_count and offer follow-up drafts if nonzero.
Do not archive everything; keep is appropriate for newsletters the user wants
to keep but has already read.
