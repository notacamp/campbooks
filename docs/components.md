# Campbooks Component Guidelines

## Why Components

All reusable UI must live in `app/components/campbooks/` as Phlex components. This keeps visual
consistency, reduces duplication, and makes UI changes predictable. Before writing HTML in a
view, check if a component already covers that pattern. If not, extract it.

## Anatomy of a Phlex Component

```ruby
# app/components/campbooks/example.rb
module Campbooks
  class Example < Campbooks::Base
    # @param variant [Symbol] :primary, :outline, :ghost
    # @param size [Symbol] :sm, :md, :lg
    def initialize(variant: :primary, size: :md, **attrs)
      @variant = variant
      @size = size
      @attrs = attrs
    end

    def view_template(&content)
      tag = @attrs[:href] ? :a : :button

      public_send(tag, class: computed_classes, **@attrs) do
        # icon slot
        yield_content(&content)
      end
    end

    private

    def computed_classes
      # Merge base, variant, and size classes
      [BASE_CLASSES, VARIANT_CLASSES[@variant], SIZE_CLASSES[@size]].join(" ")
    end
  end
end
```

## Naming Conventions

- Module: `Campboards::` prefix (e.g., `Campbooks::Button`, `Campbooks::Card`)
- File: snake_case matching class name (`button.rb`, `card.rb`)
- Rails helper: auto-generated by phlex-rails (e.g., `campbooks_button` for `Campbooks::Button`)

## Variant + Size Pattern

Use keyword args with `Symbol` values. Map variants to Tailwind classes via private constants:

```ruby
VARIANT_CLASSES = {
  primary: "bg-accent-600 text-white hover:bg-accent-700",
  outline: "border border-gray-300 text-gray-700 hover:bg-gray-50",
  ghost:   "text-gray-600 hover:bg-gray-100",
  danger:  "bg-red-600 text-white hover:bg-red-700"
}.freeze

SIZE_CLASSES = {
  sm: "px-2.5 py-1 text-xs rounded",
  md: "px-4 py-2 text-sm rounded-lg",
  lg: "px-6 py-3 text-base rounded-lg"
}.freeze
```

## Slots

For composable components (Card, Modal, PageHeader), use Phlex slots:

```ruby
class Example < Campbooks::Base
  def view_template
    render Campbooks::Card.new do |card|
      card.with_header { "Title" }
      card.with_body   { "Content" }
      card.with_footer { campbooks_button(variant: :primary) { "Save" } }
    end
  end
end
```

## When to Create a Component

- **Always**: buttons, badges, inputs, selects, checkboxes, cards, modals, avatars
- **When repeated 2+ times**: page headers, empty states, icon buttons, alerts, dividers, spinners
- **When the pattern is likely to be reused**: status indicators, folder icons, notification items

## When NOT to Create a Component

- One-off layout wrappers specific to a single page
- Patterns that differ in behavior, not just style (keep those as view partials)
- Trivial single-element wrappers (just use Tailwind directly)

## Lookbook Previews

Every component needs a Lookbook preview in `test/components/previews/` (Lookbook's default path).
The preview must show every variant and size combination.

```ruby
# spec/components/previews/example_component_preview.rb
class ExampleComponentPreview < Lookbook::Preview
  def default
    render Campbooks::Example.new(variant: :primary, size: :md)
  end

  def outline
    render Campbooks::Example.new(variant: :outline, size: :md)
  end
end
```

## Accessibility

- Buttons inside icon-only wrappers need `aria-label`
- Modals need `role="dialog"`, `aria-modal="true"`, focus trap
- Alerts need appropriate `role` (`status` for notices, `alert` for errors)
- Form inputs must have associated labels (`for` attribute or wrapping)
- Empty states should communicate status to screen readers

## Tailwind v4 Notes

- Use `@import "tailwindcss"` syntax (already configured)
- Custom theme tokens (accent-*, attention-*) are available via `@theme` in application.css
- Use the project's custom colors: `accent-600`, `accent-700`, `attention-*`
- No `@apply` in component files — compose classes in Ruby

## Responsive design

**The app must stay fully mobile responsive — no horizontal overflow down to 375px.**
Verify every UI change at mobile width, not just desktop. This is a hard requirement,
not a nice-to-have.

### Mobile-first rule

Base classes target mobile; add `sm:`/`md:`/`lg:` to *restore* the wider layout. Never
write a desktop-only layout and never drop desktop behaviour when fixing mobile.

```
grid-cols-3                  ❌ never collapses
grid-cols-1 sm:grid-cols-2 lg:grid-cols-3   ✅ collapses on mobile, full grid on desktop
```

### Patterns

- **Grids** — every multi-column grid needs a `grid-cols-1` base + responsive steps.
- **Tables** — must live in an `overflow-x-auto` wrapper so they scroll on mobile. Never
  `overflow-hidden` (it clips columns and makes them unreachable). `Campbooks::Table`
  already does this; raw `<table>`s and `Campbooks::Card(overflow: :hidden)` wrappers need
  an explicit `<div class="overflow-x-auto">`.
- **Page headers / toolbars** — `flex flex-col gap-3 sm:flex-row sm:items-center sm:justify-between`;
  give action button groups `flex-wrap`. `Campbooks::PageHeader` already stacks.
- **Side-by-side panes** — `flex flex-col lg:flex-row gap-6`; the fixed pane becomes
  `w-full lg:w-[30%]` (keep the fixed width only at the breakpoint).
- **Fixed widths** (`w-96`, `w-[480px]`, `w-[800px]`) — prefix with `w-full sm:` so they
  fill the screen on mobile (see `Campbooks::Modal`, `EmailDrawer`).
- **Wide HTML email bodies** — wrap in `overflow-x-auto` (done in `Campbooks::EmailDetail`)
  so fixed-width newsletter tables scroll within the message, not the page.

### Breakpoints used by the shells

- **Topbar** (`shared/_topbar.html.erb`): hamburger (`mobile-menu` Stimulus controller)
  below `md`; full desktop nav at `md+`; username hidden until `lg`. The switch is `md`
  (768px) because the nav + icon cluster does not fit below that.
- **Email + Scout shells** (`email_messages/{show,new,index}`, `agent_chat`): the multi-pane
  desktop split collapses to a single pane below **`lg` (1024px)** — the only width where
  folder strip + thread list + detail + Scout all fit. Helpers:
  - `email-mobile` controller — toggles list/detail panes; its `breakpoint` value is `1024`
    for email, default `640` for Scout chat.
  - `scout-mobile` controller — opens the Scout panel as a full-screen overlay on mobile.
  - `email_messages/_mobile_folder_bar` — scrollable folder chips for mobile folder switching.
- **Settings** use the layout's `content_for :sidebar` block
  (`flex flex-col gap-6 lg:flex-row`) with the sidebar `w-full lg:w-56` so it stacks above
  the content on mobile.

### Verifying

Resize and check `document.documentElement.scrollWidth > clientWidth` is false at
**375 / 640 / 768 / 1024px** (ignore `position: fixed` overlays, which manage their own
width). Use Playwright per the test-with-playwright rule. The Tailwind watcher
(`bin/rails 'tailwindcss:watch[always]'` — quote the arg in zsh) must be running for new
utility classes to compile.

## Multi-Step Modals

Setup and configuration modals must be multi-step wizards — one question per view, not a
single form cramming everything together. Seeing all fields at once overwhelms users; a
focused, conversational flow feels human and builds confidence.

### Rules

1. **One question per step.** Each step presents a single field or tightly-related group
   (e.g., city + postal code together is fine; name + description + color is not).
2. **Descriptive titles.** The modal title must describe the current step — never use a
   generic label like "Setup." Write titles that read like a friendly question or prompt.
3. **Warm, human language.** Use plain, conversational text in labels, placeholders, and
   help text. Avoid jargon and system-level terms.
4. **Animate between steps.** Use `animate-fade-in` for step transitions so the flow feels
   smooth, not jarring.
5. **Progress indicator.** Show "Step X of N" so users know where they stand.
6. **Enter advances.** Pressing Enter on a non-final step moves forward instead of
   submitting. Only the last step shows the submit button.

### Implementation

Use the `multi-step-form` Stimulus controller on the form element:

```erb
<%= form_with url: setup_path(:example), method: :patch,
      data: { controller: "multi-step-form" } do |f| %>
  <div data-multi-step-form-target="step" data-step-title="What's your organization called?">
    <%# single field here %>
  </div>
  <div data-multi-step-form-target="step" data-step-title="What does it do?">
    <%# single field here %>
  </div>
  <%# navigation buttons %>
<% end %>
```

The controller handles visibility, title updates, progress text, and next/back navigation.
Each step element declares its title via `data-step-title`.

## Notifications

One vocabulary, two surfaces. Severity is always `:success`, `:error`, `:warning`, or `:info`.

- **`:success` / `:info`** → **ephemeral toast** (bottom-right, auto-dismiss).
- **`:error` / `:warning`** → **inline alert** (`Campbooks::Alert`) at the top of `<main>`.

`notice`/`alert` remain as aliases (`notice → success`, `alert → error`) so legacy/library
flash still renders, but new code uses the canonical keys.

### From a controller (redirect / full render)

Flash types are added by the `Notifiable` concern (included in `ApplicationController`):

```ruby
redirect_to workflow_path(workflow), success: "Workflow saved."
redirect_to settings_path,           error:   "Couldn't save settings."
flash.now[:error] = "Current password is incorrect."   # on re-render (e.g. validation)
```

### From a controller (Turbo Stream, no redirect)

Use `notify_stream` to append an ephemeral toast to the toast region:

```ruby
render turbo_stream: notify_stream("Message sent!")                     # :success by default
render turbo_stream: [other_streams, notify_stream(msg, severity: :error)]
```

### Internals

- `shared/_flash_inline` (top of `<main>`) renders error/warning via `Campbooks::Alert`.
- `shared/_flash_toast_region` (fixed bottom-right) holds `#action_toasts` (ephemeral) and
  `#toasts` (the persistent notification-center toasts), stacked so they never overlap.
- The persistent notification center (`Notification` model → `Campbooks::Toast` → bell) is a
  **separate** system — don't route flash through it.

> **Migrating to RubyUI:** `Campbooks::Alert` and `Campbooks::ActionToast` are thin adapters.
> When RubyUI's `Alert`/`Toast` land, swap them behind these call sites — the controller API
> (`success:` / `error:` / `notify_stream`) stays the same.

## Design Review

When building a new component or making meaningful visual changes, invoke the `impeccable`
skill to audit for visual consistency, hierarchy, spacing, typography, and accessibility.
